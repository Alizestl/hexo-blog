0x00

此为学习PWN的记录。

It’s the initial unit.

此文从一道基础PWN题入手，记录各种细碎基础知识点。

题目 是NewStar CTF 2022的第一道PWN，题名即知识点：【ret2text】。

0x01 原理

  • Stack Overflow 栈溢出

    程序使用了未对输入数据进行检查的危险函数。攻击者可通过输入过量的数据实现攻击：即先使用垃圾数据进行填充，使得其精心构造的数据溢出覆盖指定地址，从而实现攻击者的目的。

  • ret2text

    即return to text，篡改栈帧上的返回地址为程序中已有的后门函数，如system('/bin/sh'); execve('/bin/sh', NULL, NULL);，后门函数的地址可用 IDA 或 objdump查询。

0x02 查看文件信息

checksec

查看程序开启了哪些保护机制

  • Arch：程序架构

  • RELRO：Relocation Read-Only

  • Stack：栈溢出保护，是一种缓冲区溢出攻击缓解手段。

    当函数存在缓冲区溢出攻击漏洞时，攻击者可以覆盖栈上的返回地址来让shellcode能够得到执行。当启用栈保护后，函数开始执行的时候会先往栈里插入类似cookie的信息，也就是这边的canary。当函数真正返回的时候会验证canary是否合法，如果不合法就停止程序运行。攻击者在覆盖返回地址的时候往往也会将canary给覆盖掉，导致栈保护检查失败而阻止shellcode的执行。

  • NX：no-execute，IP不会指向堆栈上的数据，即不执行堆栈上的数据。开启NX保护后，若攻击者堆栈上部署shellcode并触发，程序会直接崩溃，但可利用ROP绕过

  • PIE：Position-Independent Executable, 位置无关可执行文件。PIE技术将程序编译为位置无关，程序各个段（date segment，text segment…）加载时的虚拟地址在装载时确定，配合ASLR（Address space layout randomization）虚拟地址随机加载，使得攻击者无法确定内存布局。传统的改写GOT表失效。

这些保护机制中的设计哲学，值得日后深入研究。

  • ASLR 是如何保护 Linux 系统免受缓冲区溢出攻击的
  • 新版checksec，后期有空填坑

0x04 思路

利用__isoc99_scanf()写字符串，覆盖main函数的返回地址，让其指向system(“/bin/sh”)这个后门函数。

确定思路后，我们要解决的最后问题是计算偏移量，也就是我们需要输入多少字节的数据。

0x05 计算偏移量

IDA

一种方法是借助 IDA 。

通过在 IDA ，定位到 __isoc99_scanf("%s", v4);，跟进观察v4变量

可看到：

  • v4变量的缓冲区大小：v4[32]，即32个字节
  • v4距离rbp的偏移量：[rbp-20h]，即0x20h个字节

我们填充的数据，需要覆盖这32个字节从而溢出，再加上rbp自身所占8字节大小（32位系统是4字节，64位是8字节），我们一共需要40个字节的垃圾数据，才能覆盖到后门函数的地址。

后门函数的地址也可用 IDA 的 IDA-View查到。
最便捷的方式，就是在Function name栏双击backdoo0o0r这个函数，IDA会自动跳转到函数起始位置。
由于 IDA一打开是 Graph View视图，查找地址时先右击选择 Text View 视图，以便地址的读取。
